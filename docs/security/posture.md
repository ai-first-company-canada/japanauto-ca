# Security posture
> Captured 2026-06-11 from a code-level read of the repository. Per [DOCS-CONVENTIONS.md](../../DOCS-CONVENTIONS.md) R5, verify cited symbols against the code when relying on this document.

## Summary

JapanAuto is an Astro 6 SSG storefront fronting Cloudflare Pages Functions (D1/KV/R2) with JWT cookie auth, Cloudflare Images, and a planned Stripe billing tier. A 2026-06-09 adversarial audit raised 81 findings (0 Critical, 8 High, 29 Medium, 38 Low, 6 Info). A remediation sprint (git log c57e0a5..5ebc44b plus the 88cf5ad..dac918 security run) closed every High and nearly all of the security-relevant Medium findings, verified here against the live code under functions/. Auth, CSRF, CSP, rate limiting, input validation, IDOR/authz, listing-lifecycle integrity, and PII minimization are now genuinely wired. The payments path is the principal caveat: Stripe is intentionally still a skeleton — both functions/api/stripe/webhook.ts (no HMAC signature verification) and functions/api/boost/checkout.ts (notImplemented) are unfinished, so there is no live payment side-effect today but the webhook MUST gain fail-closed signature verification before any fulfilment UPDATE is wired. Remaining open items are low-severity cost/hardening gaps (orphan CF Images purge, unverified-email media uploads, PBKDF2 100k Workers cap, no public/_headers backstop) plus the Stripe completion. npm audit reports 0 dependency vulnerabilities.

## Controls in place

- Auth model: JWT (HS256) in HttpOnly+SameSite=Lax cookies (jc_access 15m / jc_refresh 30d). Access tokens carry a token_epoch claim re-checked against the live dealers row in middleware (functions/_middleware.ts), giving a server-side kill switch on logout/reset/suspend. importHmacKey fails closed if JWT_SECRET is missing or <32 chars; verifyAccessToken catches the throw and denies (returns null) rather than 500ing.
- Refresh-token rotation with OWASP reuse detection: presenting an already-rotated/revoked token revokes the dealer's entire token family (functions/api/auth/refresh.ts -> revokeAllRefreshTokensForDealer).
- CSRF: two-layer guard (functions/api/_lib/csrf.ts) on all /api/* unsafe methods — Sec-Fetch-Site + Origin allowlist check in _middleware.ts before any handler/D1 work, re-checked in requireDealer. Non-browser clients with no Origin/Sec-Fetch (curl, Stripe delivery) pass through since they carry no victim cookies.
- CSP: script-src carries NO 'unsafe-inline'. Inline scripts allowlisted by build-time SHA-256 hashes for SSG pages (functions/_lib/csp-script-hashes.ts) and a per-request nonce for dynamic page-shell routes. Also object-src 'none', base-uri 'self', form-action 'self', frame-ancestors 'self'. Inline on*= handlers fail the build gate. style-src still uses 'unsafe-inline' (tracked follow-up).
- Other security headers on every response: HSTS (max-age 1y, includeSubDomains, preload), X-Content-Type-Options nosniff, X-Frame-Options SAMEORIGIN, Referrer-Policy strict-origin-when-cross-origin, Permissions-Policy. CORS is per-request against an Origin allowlist (apex/www + pages.dev preview).
- Rate limiting: atomic D1-backed fixed-window limiter (functions/api/_lib/rate-limit.ts, migration 0008) using a single INSERT ... ON CONFLICT ... RETURNING serialized by SQLite's write lock — replaced the non-atomic KV read-modify-write that a parallel burst could bypass. Buckets: login per-email/min + per-email hour/day + per-IP + global anti-spray ceiling; signup per-IP; listing-create per tier; media upload-url 100/h per dealer; refresh 60/h per IP. Fails closed on D1 error.
- Input validation: zod schemas on every write. URL fields (website/promo) use a safeUrlSchema refined to http(s)-only, closing the stored-XSS vector where z.string().url() accepted javascript:/data:. Server-side HTML rendering uses esc() on all sinks and a safeUrl() helper (functions/_lib/page-shell.ts) that forces unsafe schemes to '#'. Domain validators in place: VIN checksum, NANP phone (rejects +1555), HH:MM hours, postal code, slug. Middleware enforces a 64 KB /api/* body cap and array cardinality caps guard storage amplification.
- Authorization / IDOR: PATCH /api/dealers/me uses an explicit MUTABLE_COLUMNS allowlist so type/email/slug cannot be self-mutated (mass-assignment closed). GET /api/listings/:id returns non-active/expired rows and their internal fields (flagged_reason, boost_*, dealer_id) only to the authenticated owner. Media finalize binds client-supplied image_id to the minting dealer via a pending_media_uploads table (migration 0009); upload-url tags CF uploads with dealer metadata + 30-min expiry.
- Listing lifecycle integrity: server-side state machine (LEGAL_STATUS_TRANSITIONS) with terminal-state guards; revival to 'active' forces a fresh TTL, re-runs the rolling year-window age cap, and clears any resurrected boost. Read side filters past-TTL rows off every public surface; a standalone cron Worker (workers/expire-sweeper, every 6h) flips status='active' rows past expires_at to 'expired'.
- PII handling: refresh_tokens stores a stable-salted SHA-256 IP hash (hashIpStable) instead of raw IP; contact_reveals uses a daily-salted IP hash. Passwords: PBKDF2-HMAC-SHA256, 100k iterations (Workers WebCrypto hard cap), 16-byte per-user salt, constant-time compare.
- Infra/CI: GitHub Actions deploy.yml now runs the SEO audit gate (setup-python + npm run audit:seo) between build and deploy, matching the local predeploy hook; branch name passed via a quoted env var instead of interpolated into the shell (branch-name injection closed). npm audit = 0 vulnerabilities.
