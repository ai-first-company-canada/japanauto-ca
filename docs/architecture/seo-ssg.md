# SEO, SSG & launch gating
> Captured 2026-06-11; updated 2026-07-05 post deep-audit remediation (commit cf5de1f: ADV-1 catalog-freshness model, WEB-1 six new legal/info pages, WEB-5 internal-link-integrity gate). Per [DOCS-CONVENTIONS.md](../../DOCS-CONVENTIONS.md) R5, verify cited symbols against the code when relying on this document.

## Purpose

The build-time SEO/GEO surface for japanauto.ca: ~925 city-first Astro SSG pages (incl. six top-level legal/info pages added 2026-07-05 — terms/privacy/cookies/accessibility/about/cities) plus D1-backed dynamic detail/sitemap Pages Functions, every indexable page carrying title/description/self-canonical/single-H1/Open Graph/JSON-LD, enforced by a Python audit that gates deploy and a stricter LAUNCH mode that refuses to ship fabricated demo inventory, dead internal links, or a host-blocking robots.txt before the production domain is connected. The browse (city/model) surface is a build-time snapshot of prod D1 refreshed on every deploy and by a 3-hourly CI schedule; listing detail pages are live D1 Functions (see "Catalog freshness" below).

## How it works

SSG generation: astro.config.mjs sets site=https://japanauto.ca, build.inlineStylesheets='always', and a remark-strip-faq plugin. Page templates under src/pages/** use getStaticPaths() (e.g. [city]/[make]/[model]/index.astro iterates TIER_1_CITIES x Object.keys(MODELS_BY_BRAND) x models) to emit ~925 static HTML files (incl. the six top-level legal/info pages added 2026-07-05). Every page renders BaseLayout with title/description/canonical (always the literal https://japanauto.ca/<path>/ — city baked in, no client fetch, no data-geo) and an optional schemaLD array built from src/lib/schema-graph.ts helpers (breadcrumbList/itemList/faqPage) or inline literals.\n\nJSON-LD: BaseLayout.astro always emits a root @graph with Organization(#organization, knowsAbout entity list, contactPoint) + WebSite(#website, inLanguage en-CA). A SearchAction was deliberately removed (no /search endpoint — same anti-pattern faqPage() guards). Page schemaLD is spread into @graph (lines 29-74). The JSON-LD <script> escapes '<' to \\u003c. Dynamic routes replicate this via functions/_lib/page-shell.ts renderShell() (a leaner Organization+WebSite graph).\n\nDemo gating (audit #6): the isDemo mechanism is retained as a guard, but since the 2026-06-12 live-data wiring catalog-stubs.ts getCatalogForModelCity() always returns isDemo:false (rows come from catalog-live.json / prod D1, empty state when none). In [city]/[make]/[model]/index.astro the Vehicle/Offer ItemList (lines 79-103) and the featured Offer (114-121) are spread only when !catalog.isDemo; the historic isDemo branch would render a data-demo-content='true' 'Sample preview' banner (172-182). With isDemo false and prod D1 empty, no such banner/marker is emitted and Offer/Vehicle markup is simply absent (no live rows to mark up), so structured data for nonexistent offers is never emitted — the guard stands ready for the day the flag ever becomes per-row.\n\nNoindex surfaces: AuthLayout.astro and DealerLayout.astro emit <meta name=robots content='noindex, nofollow'>; the audit treats those plus 404.html as non-indexable and skips hard checks (seo-audit.py main() idx flag, line 153).\n\nThe gate: predeploy (package.json) runs typecheck -> build (astro build + extract-css + generate-csp-hashes) -> audit:seo. seo-audit.py scan() regex-extracts title/desc/canonical/robots/h1-count/OG-presence/JSON-LD @types/word-count/data-demo-content per file; main() hard-errors per indexable page on the six invariants and prints a per-group coverage table. LAUNCH=1 (audit:launch) adds four launch-only hard-error families: (a) any page carrying data-demo-content (capped at 5 listed + a '+N more' line); (b) a fabricated-literal blocklist (invented dealer/junkyard names + specific fake counts left from the demo era that carry no data-demo-content marker); (c) **internal-link integrity (deep-audit WEB-5)** — every `href="/…"` (excluding `/api/`, `/dealer/`, `/dashboard` runtime/auth paths) must resolve to a built page, a top-level dist asset, or a `public/_redirects` rule (named params `:make` etc. become `[^/]+`); an unresolved link is a dead link that WILL 404 the instant indexing opens, so it blocks deploy; (d) robots.txt/sitemap.xml presence + a not-`^Disallow:\\s*/\\s*$` robots check. WEB-5 was the missing control that would have caught the WEB-1 dead footer/nav links and the WEB-3 unauthored blog links; on first run it immediately caught 3 more (an EV blog post linking non-catalog models), now repointed to brand hubs.\n\nCrawl/index posture: production public/robots.txt ships fully OPEN; the only thing keeping *.pages.dev staging out of indexes is functions/_middleware.ts returning a Disallow:/ robots + X-Robots-Tag noindex for preview hostnames. Connecting japanauto.ca therefore = instant indexing (per memory + LAUNCH-CHECKLIST), which is exactly why the LAUNCH gate exists. Sitemaps: /sitemap.xml (index) -> sitemap-static.xml (build-time, ~887 URLs) + sitemap-listings.xml + sitemap-donors.xml (Pages Functions, D1, 1h cache, currently empty urlsets while D1 is empty — correct).

Catalog freshness (deep-audit ADV-1): the SEO surface splits into two consistency models. (1) Browse/city/model pages are a build-time SSG SNAPSHOT of prod D1 — `getStaticPaths()` builds them from `src/data/catalog-stubs.ts` over `src/data/catalog-live.json`, which `scripts/export-catalog-data.mjs` re-exports (`WHERE l.status='active'`) at deploy time. So browse inventory + every "N listed" count only advance when a build runs. Before ADV-1 that meant a dealer's new listing stayed orphaned from all browse pages/counts until an operator manually redeployed. Now the snapshot is refreshed on EVERY deploy (local predeploy AND CI "Refresh catalog from D1" step) and additionally by a 3-hourly CI `schedule:` cron (`.github/workflows/deploy.yml`, `17 */3 * * *`), so browse pages track D1 within ~3h with no manual redeploy. The refresh is fail-safe: if the deploy token lacks `D1:read` on japanauto-prod, the export fails soft and the build proceeds from the committed snapshot with a CI warning (details in docs/architecture/infra-ops.md). (2) Listing DETAIL pages are already live — `functions/used-cars/listing/[slug].ts` reads the real D1 row per request (404/expiry enforced read-side) and each newly-active listing is IndexNow-pinged instantly. So a new listing's detail page is live + submitted immediately; only its appearance in browse/counts is snapshot-bound and now bounded to the 3-hourly interval. (Owner residual per the run's NEW-GATE-2: confirm the browse-appearance step live end-to-end with a throwaway dealer.)

## Invariants

- Every indexable page (not noindex, not 404) must have: a <title>, a meta description, a <link rel=canonical> whose href === the page's own production URL (https://japanauto.ca/<path>/), exactly one <h1>, all four og: tags (title/description/image/url), and at least one valid JSON-LD block. seo-audit.py hard-fails otherwise and the predeploy/CI gate blocks the deploy.
- Canonical, og:url, and the JSON-LD breadcrumb leaf must all be the city-first literal URL — city is baked into HTML at SSG time. No data-geo* attributes, no client-side geo fetch, no edge HTML rewriting (the audit only sees the literal built HTML).
- Vehicle/Offer (and featured Offer) JSON-LD must NOT be emitted while listings are demo stubs (catalog.isDemo). Marking up nonexistent offers is a Google manual-action risk; the gate is !catalog.isDemo in [city]/[make]/[model]/index.astro.
- FAQPage JSON-LD is only emitted for Q&A pairs that have a real, visible answer — faqPage()/faqFromBody() drop placeholder/empty answers, and the audit's faq_visible-without-FAQPage finding is a WARNING (legitimate when answers are 'coming soon'), never a hard error.
- Demo/fabricated content must be machine-detectable: any stubbed inventory page must stamp data-demo-content into HTML so LAUNCH mode can refuse to ship it.
- The production domain must not be connected until LAUNCH mode is green — connecting japanauto.ca opens indexing immediately, and public/robots.txt is already open. The Disallow:/ protection exists only at runtime for *.pages.dev, not in the shipped robots.txt.
- BaseLayout/page-shell own the Organization + WebSite root @graph; page templates must never re-emit Organization/WebSite (they only supply page-specific nodes via schemaLD).
- sitemap-listings/donors must only list status='active', non-expired rows (read-side TTL enforcement; expire-sweeper is best-effort).
- Declared structured-data actions must be backed by a live endpoint — SearchAction was removed because /search doesn't exist; re-add only when it ships.
- Every static internal link must resolve (deep-audit WEB-5): a footer/nav/body `href="/…"` that isn't a built page, a top-level dist asset, or a `public/_redirects` rule is a hard 404 the moment indexing opens, and audit:launch blocks the deploy on it. When you remove a page, remove its links or add a redirect; when you link a page, author it or redirect it. `/api/`, `/dealer/`, `/dashboard` are excluded (runtime Functions / auth-gated, not static pages).
- Catalog freshness is two-tier: listing DETAIL pages are live D1 Functions (instant + IndexNow-pinged); browse/city/model pages + counts are a build-time snapshot refreshed only by a deploy or the 3-hourly CI schedule. Never assume a just-created listing is on its browse page immediately — it is bounded to the ADV-1 refresh interval, and only if the deploy token holds `D1:read`.

## Design decisions

- **Validate SEO/GEO against the built dist/ HTML with a standalone Python regex scanner, run as a predeploy hook and a separate CI step, that hard-fails the deploy.** — The audit must check the exact bytes crawlers and answer engines receive, after Astro rendering, not the source templates. Making it a deploy gate means SEO-broken HTML cannot ship. CI re-runs it because wrangler-action calls 'pages deploy' directly and bypasses the npm predeploy lifecycle hook (audit #19). *Rejected:* An Astro integration / unit tests on component output (rejected — wouldn't see final merged HTML, JSON-LD escaping, or inline-style/marker output); a third-party SEO crawler in CI (heavier, network-dependent, no deploy-blocking semantics).
- **Gate Vehicle/Offer/featured-Offer JSON-LD and the demo banner on a single isDemo boolean rather than emitting markup for stub listings (audit #6).** — Stub catalog rows have URLs that don't resolve and prices that are fabricated; emitting Offer/Vehicle markup for them is structured-data-for-nonexistent-offers (Google manual-action territory). One flag keeps the visible 'Sample preview' banner, the data-demo-content marker, and the schema suppression in lockstep, and makes the launch gate trivial. *Rejected:* Always emit Vehicle/Offer and rely on later cleanup (rejected — risks a manual action the moment the site is indexed); per-row honesty flags now (deferred until the D1 query replaces the stub — comment says 'must become false or derived per-row').
- **Ship public/robots.txt fully open and keep staging out of indexes purely at runtime via middleware (Disallow:/ + X-Robots-Tag only for *.pages.dev), backed by a LAUNCH audit.** — There is no manual 'open robots' step at cutover — connecting the domain must Just Work for indexing. The risk (indexing fabricated/demo data) is moved entirely into the LAUNCH gate, which refuses to build while data-demo-content or a blocking dist/robots.txt is present. *Rejected:* Ship a blocking robots.txt and remember to flip it at launch (rejected — error-prone manual step, and a forgotten flip silently blocks the whole site); environment-specific robots files (more moving parts than a single host-sniffing middleware branch).
- **Two-tier catalog freshness: browse/city/model pages are a build-time D1 snapshot refreshed on every deploy + a 3-hourly CI schedule; listing detail pages are live D1 Functions (deep-audit ADV-1).** — Keeping the ~925 browse pages fully static preserves the SSG performance/cache posture and lets the launch SEO gate inspect exact built bytes, but a pure build-time snapshot left a new listing orphaned from every browse page/count until a manual redeploy (the ADV-1 blocker — the paying partner's discoverability silently undelivered). Rather than render browse rows dynamically (losing the static cache) the fix schedules the existing export+redeploy: `deploy.yml` gained a `schedule:` cron (`17 */3 * * *`) plus a fail-safe "Refresh catalog from D1" step, so browse inventory tracks D1 within ~3h with no operator action. Detail pages were already live (Functions read D1 per request + IndexNow-ping on activation), so the honesty gap was only the browse grid. *Rejected:* Runtime-render every browse page from D1 (loses static caching, adds a D1 read to the hottest pages); leave publish=manual-redeploy (rejected — the orphaned-listing blocker); event-driven redeploy on each activation (deferred — a fixed 3-hourly cadence is simpler and bounds Pages deploy spend). Trade-off: browse freshness is bounded to the interval and depends on the deploy token holding `D1:read` (else fail-safe to the committed snapshot).
- **FAQ answers authored as markdown body prose under '## Common questions', stripped from rendered output by remark-strip-faq and lifted into a <details> accordion + FAQPage via faqFromBody().** — Keeps the FAQPage schema text byte-identical to visible accordion text (Google requires markup to mirror visible content) from a single source of truth, while preserving internal links/bold in the accordion via mdToInlineHtml. *Rejected:* Duplicating Q&A in frontmatter and body (drift risk, and the audit/Google penalize mismatched markup); rendering the prose normally and separately hardcoding schema (double-maintenance).
- **FAQPage-without-visible-FAQ, off-range title/description length, and thin (<200 word) content are WARNINGS, not deploy-blocking errors.** — These are editorial judgement calls, not correctness bugs — e.g. a visible FAQ with placeholder answers SHOULD omit FAQPage. Blocking on them would either force bad markup or stall deploys on subjective thresholds. *Rejected:* Hard-failing on length/thin-content (rejected — too brittle for a content site with legitimate short pages); ignoring them entirely (rejected — they're useful drift signals, surfaced as a counted warning summary).

## Security notes

- CI passes the deploy branch via env DEPLOY_BRANCH instead of interpolating ${{ github.ref_name }} into the shell command (audit #37) — a branch named e.g. fix/x$(...) would otherwise be command-injected on the runner that holds CLOUDFLARE_* secrets.
- JSON-LD output escapes '<' to < in both BaseLayout.astro (set:html) and page-shell.ts renderShell() to prevent </script> breakout / stored-XSS via schema fields.
- generate-csp-hashes.mjs explicitly skips non-executable script types (ld+json) when hashing and fails the build on any inline on*= handler, so the CSP can drop 'unsafe-inline'; the JSON-LD block is unaffected. The async-font-CSS rel-flip script is byte-identical across the three layouts so it dedupes to one CSP hash.
- page-shell.ts (dynamic routes) provides esc() for HTML-context output and safeUrl() that whitelists http/https/mailto/tel and rejects javascript:/data:/vbscript: (incl. tab/newline-obfuscated) before placing user/D1 data into href attributes.
- robots.txt explicitly Disallows /dealer /api /auth /dashboard; /dealer/* is additionally noindex,nofollow at the layout level and JWT-gated in middleware — defense in depth so the authenticated portal is never an SEO surface.
- Preview-host guard (middleware) returns X-Robots-Tag noindex + Disallow:/ for *.pages.dev so staging with demo data cannot be indexed even though the shipped robots.txt is open.
