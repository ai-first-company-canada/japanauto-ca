# Infra, deploy & ops
> Captured 2026-06-11 from a code-level read of the repository. Per [DOCS-CONVENTIONS.md](../../DOCS-CONVENTIONS.md) R5, verify cited symbols against the code when relying on this document.

## Purpose

Defines how JapanAuto is configured, built, gated, and shipped to Cloudflare Pages (Astro SSG + Pages Functions over D1/KV/R2/Images), plus the out-of-band cron Worker and operational concerns (security headers/CSP, rate limiting, IndexNow, migrations, launch gating).

## How it works

Two independent deploy paths target one Cloudflare Pages project (japanauto). (1) Local: `npm run deploy` triggers the npm `predeploy` lifecycle hook = `npm run typecheck && npm run build && npm run audit:seo`, then `wrangler pages deploy dist --project-name japanauto`. `build` is `astro build` (pure SSG; site=japanauto.ca) followed by `scripts/extract-css.mjs` (lifts Astro-inlined CSS to dist/styles/global.css for dynamic routes) and `scripts/generate-csp-hashes.mjs` (hashes inline scripts into functions/_lib/csp-script-hashes.ts and acts as a hard gate). `scripts/seo-audit.py dist` walks built HTML and exits non-zero on missing on-page SEO signals. (2) CI: `.github/workflows/deploy.yml` runs on push to main/feature/**/fix/** and reruns typecheck+build+audit:seo (the SEO gate is duplicated because cloudflare/wrangler-action calls `pages deploy` directly and bypasses the npm predeploy hook — audit #19), then deploys via wrangler-action using `CLOUDFLARE_API_TOKEN`/`CLOUDFLARE_ACCOUNT_ID` GitHub secrets, passing the branch name through the DEPLOY_BRANCH env var (not shell interpolation) to avoid command injection from crafted branch names (audit #37). Runtime: Pages auto-discovers `functions/` (route handlers) and `functions/_middleware.ts`. Bindings (DB/CACHE/RATE_LIMIT/MEDIA) come from wrangler.toml top-level config; the IMAGES binding is declared in env.d.ts but is NOT referenced by any function — image transforms instead go through the Cloudflare Images REST API (CLOUDFLARE_IMAGES_API_TOKEN, used in functions/api/media/upload-url.ts) and direct imagedelivery.net delivery. `onRequest` in _middleware.ts is the central ops chokepoint: it serves a Disallow-all robots.txt for *.pages.dev hosts, runs CSRF + 64KB body guards on /api/*, generates a per-request CSP nonce for non-API requests, enforces /dealer/* auth (verifyAccessToken + a token_epoch kill-switch via getDealerById), handles CORS preflight, wraps next() in a try/catch error boundary (audit #23), and finally stamps SECURITY_HEADERS + the CSP header (CSP_STATIC for SSG, nonce variant only when a dynamic route consumed the nonce). The separate cron Worker workers/expire-sweeper runs `scheduled()` every 6h (Pages Functions cannot run scheduled handlers) and flips TTL-expired active listings to 'expired'; it is deployed out-of-band and shares the prod D1 (remote=true). Rate limiting is D1-backed (rate_limits table, atomic INSERT...ON CONFLICT...RETURNING) after a non-atomic KV version was replaced (migration 0008). IndexNow pings fire fire-and-forget from API handlers when INDEXNOW_KEY is set; sitemap-listings/donors are rendered live from D1.

## Invariants

- The expire-sweeper Worker is NOT deployed by `npm run deploy`/CI (which only ship dist/ + functions/); it must be deployed separately with `cd workers/expire-sweeper && npx wrangler deploy` after any change.
- CSP script-src carries NO 'unsafe-inline'. Every executable inline script must be either build-time-hashed (SSG, via generate-csp-hashes.mjs) or nonce-stamped (dynamic, via takeCspNonce). An inline on*= event-handler attribute anywhere in dist FAILS the build — never add one.
- Unique inline-script hash count must stay <= MAX_HASHES (64); a blowout means a component is interpolating per-page values into an inline script and must be refactored to data-* attributes.
- seo-audit.py (audit:seo) must pass before any deploy; CI duplicates it because wrangler-action bypasses the npm predeploy hook. audit:launch (LAUNCH=1) must additionally be green before the production domain is attached.
- Attaching the japanauto.ca custom domain opens indexing immediately — public/robots.txt is already Allow-all; the Disallow-all robots is only synthesized by middleware for *.pages.dev. All demo/test data must be removed before domain attach.
- Branch names must reach wrangler-action via the DEPLOY_BRANCH env var, never spliced directly into the command string (injection guard, audit #37).
- JWT_SECRET must be a string >= 32 chars; auth fails closed otherwise (verifyAccessToken denies all tokens, signing throws) — a prod deploy without a valid secret breaks auth entirely rather than silently bypassing it.
- D1 database_id b0d65b95-2f43-403d-9237-0d4cac6e186a (japanauto-prod) is referenced in BOTH wrangler.toml and workers/expire-sweeper/wrangler.toml and must stay in sync.
- Rate-limit checks fail closed: a D1 error propagates to a 5xx rather than allowing the request.
- Public-by-design vars (CLOUDFLARE_ACCOUNT_ID, PUBLIC_CLOUDFLARE_ACCOUNT_HASH, INDEXNOW_KEY) may live in wrangler.toml; true secrets (JWT_SECRET, STRIPE_*, RESEND_API_KEY, DAILY_IP_HASH_SALT, CLOUDFLARE_IMAGES_API_TOKEN) must only be set via wrangler pages secret put / dashboard and never committed.

## Design decisions

- **Run the TTL-expiry sweep as a standalone cron Worker (workers/expire-sweeper) instead of inside the Pages project.** — Cloudflare Pages Functions cannot register scheduled() handlers; a cron trigger requires a real Worker. It shares the same prod D1 and the sweep is idempotent. *Rejected:* Doing expiry lazily on read only (rejected: read-side already filters expired rows, but status stayed 'active' forever, skewing dealer dashboards and analytics — audit #8). A Durable Object alarm (heavier than needed).
- **Drop 'unsafe-inline' from CSP script-src and allowlist inline scripts by build-time SHA-256 hash (SSG) plus per-request nonce (dynamic HTML).** — Eliminates the broadest XSS vector while still allowing Astro's inlined scripts; ~24 unique hashes cover 900+ static pages, and the build gate prevents regressions (on*= handlers, hash-count blowups). *Rejected:* Keeping 'unsafe-inline' (rejected: defeats CSP). Using only a nonce everywhere (rejected: SSG pages are statically cached and cannot carry a per-request nonce). 'unsafe-hashes' for event handlers (rejected: handlers refactored to addEventListener instead).
- **Duplicate the SEO audit as an explicit CI step rather than relying on the npm predeploy hook.** — cloudflare/wrangler-action invokes `pages deploy` directly and never runs the npm `predeploy` lifecycle hook, so CI could ship SEO-broken HTML a local `npm run deploy` would reject (audit #19). *Rejected:* Trusting the predeploy hook in CI (rejected: bypassed by wrangler-action). Making deploy a single npm script in CI (not done).
- **Move rate-limit counters from KV to D1 (table rate_limits, migration 0008) using atomic INSERT...ON CONFLICT...RETURNING.** — The KV version did get->put with no compare-and-swap, so parallel bursts all read the same count and bypassed limits. SQLite's write lock serializes check-and-increment, giving each concurrent request a distinct post-increment count. *Rejected:* KV with optimistic concurrency (rejected: KV has no CAS). A Durable Object counter (rejected: heavier; D1 already present). Cloudflare's native Rate Limiting (not adopted).
- **Keep the IMAGES Workers binding declared in env.d.ts but route all image work through the Cloudflare Images REST API + imagedelivery.net.** — The wrangler.toml [images] block is unsupported by the Pages CLI (must be set in the dashboard); upload-url mints Direct Upload URLs via CLOUDFLARE_IMAGES_API_TOKEN and browsers fetch variants from imagedelivery.net/<hash>. *Rejected:* Using the runtime IMAGES binding for transforms (declared but currently unwired — effectively a skeleton).
- **Keep public-by-design identifiers (account id, delivery hash, IndexNow key) in wrangler.toml [vars]; keep real secrets out, set via wrangler secret/dashboard.** — These values already appear in dashboard URLs, every imagedelivery.net URL, or the publicly hosted /<key>.txt, so committing them leaks nothing; true secrets stay out of the repo. *Rejected:* Treating everything as a secret (rejected: needless friction, and build-time inlining of PUBLIC_* needs them at build).

## Security notes

- Secrets are documented only as comments in wrangler.toml and typed in env.d.ts; actual values come from `wrangler pages secret put` / Pages dashboard and GitHub Actions secrets (CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID). .gitignore excludes .env*, .dev.vars*, .wrangler/.
- wrangler.toml has 0600 perms but contains live public identifiers (CLOUDFLARE_ACCOUNT_ID, account hash, INDEXNOW_KEY, prod D1 id) — public-by-design but it does expose infra topology if the repo is shared.
- CI deploy.yml deploys on push to main AND feature/**/fix/** branches; any branch push with repo write access ships to the Pages project (preview deployment for non-main, but still uses prod CLOUDFLARE_API_TOKEN). Branch-name injection into the deploy command was closed (audit #37) by passing via DEPLOY_BRANCH env.
- Middleware enforces HSTS preload, nosniff, X-Frame-Options SAMEORIGIN, frame-ancestors 'self', object-src 'none', a strict CSP, a 64KB /api body cap (audit #33), CSRF rejection of cross-site unsafe-method requests (audit #9), and a /dealer JWT auth guard with a token_epoch session kill-switch (audit #11). The error boundary (audit #23) ensures even thrown handlers get security headers + clean JSON.
- JWT_SECRET fails closed below 32 chars (audit #12): misconfiguration disables auth entirely instead of accepting weakly/unsigned tokens. JWT_SECRET is also reused as the stable salt for hashIpStable (one-way SHA-256, never exposes the secret).
- PII minimization: contact_reveals.ip_hash uses a daily-rotating salt (hashIp); refresh_tokens store hashIpStable (no raw IP) so a D1/backup leak never exposes plaintext dealer IP history (audit #20, PIPEDA/GDPR).
- expire-sweeper and `wrangler dev --test-scheduled` operate against the REAL prod D1 (remote=true); there is no local-simulator isolation for this Worker — destructive testing risk, mitigated only by the sweep being idempotent.
- The MEMORY note 'git push does NOT auto-deploy; use npm run deploy' is in tension with the present .github/workflows/deploy.yml, which DOES deploy on push via wrangler-action. Whether pushes actually deploy depends on the CLOUDFLARE_API_TOKEN secret being configured in the GitHub repo — verify before assuming either behavior.

## Gaps / TODO

- No periodic data-retention/cleanup job exists yet. LAUNCH-CHECKLIST §3b flags it as a TODO (delete expired/revoked refresh_tokens, used/expired verification_tokens, old contact_reveals via a Cloudflare Cron Worker — Pages Functions can't cron). Currently manual-only.
- The IMAGES runtime binding is declared in types/env.d.ts but referenced by zero functions — effectively a skeleton/unused binding; image work goes through the REST API + imagedelivery.net instead.
- No migration journal/meta directory is present under migrations/ (no _journal.json); migration numbering has a prod gap (0005 -> 0007) because 0006 is a dev-only seed kept in migrations/dev/. Per MEMORY, prod migrations are sometimes applied out-of-band, so journal-vs-schema must be verified before `migrations apply`.
- migrations/README.md is stale/inaccurate: it documents only 0001-0002, uses database_name="japanauto" (actual is japanauto-prod) and an old Obsidian-vault migrations_dir path, and its 'planned next migrations' (subscriptions_audit, email_log, listing_revisions) do not match what shipped (0003-0010).
- The CSP style-src still keeps 'unsafe-inline' (Astro inlines all stylesheets + style="" attributes); hashing styles is an acknowledged follow-up in _middleware.ts.
- No automated rollback mechanism is documented; rollback relies on Cloudflare Pages deployment history. There is no scheduled job to deploy/verify the expire-sweeper, and 'no way to manually fire a deployed cron' (only --test-scheduled against prod D1).
- Several pre-launch SEO tails are still open (LAUNCH-CHECKLIST §3): Organization logo points to a nonexistent /logo.png (only logo.svg exists), breadcrumb listing-detail links a legacy 301 URL.
- Local dev requires JWT_SECRET passed via --binding (no .dev.vars checked in), so a fresh clone cannot exercise auth without manual setup (per MEMORY).
