## Purpose

Defines how JapanAuto is configured, built, gated, and shipped to Cloudflare Pages (Astro SSG + Pages Functions over D1/KV/R2/Images), plus the two out-of-band Workers — the multi-job cron Worker (`workers/expire-sweeper`) and the Access-gated admin Worker (`workers/admin`) — and operational concerns (security headers/CSP, rate limiting, IndexNow, migrations, launch gating, secrets inventory).

## How it works

**Deploy paths.** Two independent paths target one Cloudflare Pages project (japanauto). (1) Local: `npm run deploy` triggers the npm `predeploy` lifecycle hook = `node scripts/export-catalog-data.mjs && npm run typecheck && npm run build && npm run audit:seo && npm run audit:launch`, then `wrangler pages deploy dist --project-name japanauto`. The catalog export refreshes src/data/catalog-live.json from prod D1 (so every deploy ships current SSG catalog pages); `build` is `astro build` (pure SSG; site=japanauto.ca) followed by `scripts/extract-css.mjs` (lifts Astro-inlined CSS to dist/styles/global.css for dynamic routes) and `scripts/generate-csp-hashes.mjs` (hashes inline scripts into functions/_lib/csp-script-hashes.ts and acts as a hard gate). `scripts/seo-audit.py dist` walks built HTML and exits non-zero on missing on-page SEO signals; the LAUNCH=1 pass additionally rejects demo-content markers and a closed robots.txt. (2) CI: `.github/workflows/deploy.yml` triggers on push to `main` ONLY, on a `schedule:` cron `17 */3 * * *`, and on manual `workflow_dispatch`. The old `feature/**`+`fix/**` push triggers were REMOVED (deep-audit OPS-1): a Pages preview deployment's Functions bind the PRODUCTION D1/KV/R2 (bindings are top-level in wrangler.toml, not per-environment) and the public `*.pages.dev` host exposed unauthenticated prod-D1 write paths (`track-contact`), so shipping an arbitrary branch to a public preview URL was a prod-write surface — use `workflow_dispatch` for an intentional one-off branch deploy. The job runs typecheck → **Refresh catalog from D1** → build → audit:seo → audit:launch → wrangler-action `pages deploy`. The SEO/launch gates are duplicated because cloudflare/wrangler-action calls `pages deploy` directly and bypasses the npm predeploy hook (audit #19); deploy uses `CLOUDFLARE_API_TOKEN`/`CLOUDFLARE_ACCOUNT_ID` GitHub secrets and passes the branch name through the DEPLOY_BRANCH env var (not shell interpolation) to avoid command injection from crafted branch names (audit #37). Unlike before, CI now DOES run export-catalog-data.mjs (the "Refresh catalog from D1" step, deep-audit ADV-1) so a scheduled or `main`-push build re-exports current inventory; that step is fail-safe (`node scripts/export-catalog-data.mjs || echo "::warning::…"`), so if the deploy token lacks `D1:read` on japanauto-prod (or D1 is unreachable) the build falls back to the committed catalog-live.json snapshot and only warns — a stale-but-honest catalog beats a broken deploy.

The `schedule:` cron exists because the SSG browse/city/model pages are a build-time snapshot of prod D1 (catalog-live.json): without a periodic rebuild a dealer's newly-active listing is live on its detail page + IndexNow-pinged but ORPHANED from every browse page and count until someone redeploys (the ADV-1 blocker). The 3-hourly rebuild re-exports from D1 and redeploys so browse inventory + counts track D1 within the interval with no manual redeploy; `17 */3 * * *` ≈ 240 runs/month, tuned to stay under the Pages deploy budget. Detail pages are already instant (live Functions from D1). The refresh only closes the gap if the deploy token holds `D1:read`; otherwise every scheduled run just redeploys the committed snapshot with a CI warning.

**Pages runtime.** Pages auto-discovers `functions/` (route handlers) and `functions/_middleware.ts`. Bindings (DB/CACHE/RATE_LIMIT/MEDIA) come from wrangler.toml top-level config; the IMAGES binding is declared in env.d.ts but is NOT referenced by any function — image transforms instead go through the Cloudflare Images REST API (CLOUDFLARE_IMAGES_API_TOKEN, used in functions/api/media/upload-url.ts) and direct imagedelivery.net delivery. `onRequest` in _middleware.ts is the central ops chokepoint: it serves a Disallow-all robots.txt for *.pages.dev hosts, runs CSRF + 64KB body guards on /api/*, generates a per-request CSP nonce for non-API requests, enforces /dealer/* auth (verifyAccessToken + a token_epoch kill-switch via getDealerById), handles CORS preflight, wraps next() in a try/catch error boundary (audit #23), and finally stamps SECURITY_HEADERS + the CSP header (CSP_STATIC for SSG, nonce variant only when a dynamic route consumed the nonce). Rate limiting is D1-backed (rate_limits table, atomic INSERT...ON CONFLICT...RETURNING) after a non-atomic KV version was replaced (migration 0008). IndexNow pings fire fire-and-forget from API handlers when INDEXNOW_KEY is set; sitemap-listings/donors are rendered live from D1.

**Cron Worker (workers/expire-sweeper) — four scheduled jobs.** Pages Functions cannot register scheduled() handlers, so all recurring jobs live in this one standalone Worker (the name is historic), deployed out-of-band and sharing prod D1 (remote=true). `scheduled()` in src/index.ts dispatches on an exact match of `controller.cron` against the four trigger strings and throws on anything else (a trigger reformatted in the dashboard fails loudly instead of running the wrong job):

| Cron (UTC) | Dispatch | What it does |
|---|---|---|
| `0 */6 * * *` (SWEEP_CRON) | `sweepExpired()` | Flips TTL-expired `status='active'` listings to `'expired'` (idempotent; second half of audit #8), then a retention batch (LAUNCH-CHECKLIST §3b, closed 2026-06-12): DELETE refresh_tokens >30 days past expiry, consumed/expired verification_tokens, contact_reveals >90 days, rate_limits windows >48h (2× the largest 24h window — keys embed raw IPs/emails); UPDATE featured_slots past active_until to `'ended'` so the (city, make) exclusivity pair frees on non-payment (ADR-0013). |
| `45 9 * * *` (MARKET_SYNC_CRON) | `syncMarketStats()` | Pulls the scraper project's `japanauto_market_stats` Supabase view via PostgREST (deterministic order over the full PK, 1000-row pages) and replaces the D1 `market_stats` snapshot: INSERT OR REPLACE in 6-row statements (D1's 100-bound-param cap), then DELETE rows with stale synced_at. Auth ladder, tried in order until one authenticates: (1) `jwt-role (least privilege)` — sb-secret apikey + the japanauto_sync HS256 JWT in Authorization; (2) `legacy anon+jwt`; (3) `sb-secret only` (service-role rights — fallback because the scraper project runs ES256 signing keys). 401/403 on a rung logs loudly and falls one rung; all rungs rejected → throw. 0 rows from the view → keep yesterday's snapshot. Rows with unknown mileage_bucket/seller_kind are skipped-and-logged, not batch-aborted. View emits whole CAD dollars; the sync stores INTEGER cents (×100). |
| `0 14 * * 1` (WEEKLY_REPORT_CRON) | `sendReports(env, "weekly")` | Monday dealer e-mail reports covering the previous 7 days (period anchored to the most recent Monday so a late re-run dedupes, not double-sends). |
| `30 14 1 * *` (MONTHLY_REPORT_CRON) | `sendReports(env, "monthly")` | Monthly reports on the 1st covering the previous calendar month. |

Reports (src/reports.ts, decision 0016) go to every dealer with `reports_opt_out = 0`, all tiers: free dealers get own stats + an honest Pro teaser computed from real sold data (`proTeaser`, sample ≥3 or a number-free pitch — fabricated numbers are banned project-wide); pro/trial dealers additionally get market position per active lot (market_stats, `seller_kind='dealer'` benchmark per migration 0019) and the social/paid traffic split (`entity_stats_daily.views_social/views_paid`, migration 0018). Idempotency: `INSERT OR IGNORE INTO report_runs (period, dealer_id, …)` reserves the slot before send; a failed or empty send releases the reservation. Delivery is the Resend HTTP API; missing RESEND_API_KEY/REPORTS_UNSUB_SECRET → logged no-op. Every mail carries a CASL one-click unsubscribe URL `…/api/reports/unsubscribe?d=<dealer_id>&s=<hmac>` minted by the Worker and verified by the Pages function — the HMAC secret must be identical on both deployables — plus the s.6(2) sender line from the REPORTS_SENDER_LINE var.

**Admin Worker (workers/admin) — admin.japanauto.ca behind Cloudflare Access.** A second standalone Worker (decision 0014) so ZERO admin code ships in the public Pages bundle; routed via `custom_domain` to admin.japanauto.ca, `preview_urls = false` (version-preview hostnames would sit OUTSIDE the Access application). Layers: Cloudflare Access (Zero Trust, team `japanauto`, OTP policy pinned to ADMIN_EMAILS) terminates unauthenticated traffic at the edge; `requireAdmin()` (src/lib/access.ts) independently re-verifies the `Cf-Access-Jwt-Assertion` RS256 JWT against the team JWKS and checks alg, kid, signature, exp, iss (`https://japanauto.cloudflareaccess.com`), aud (pinned ACCESS_AUD) and the email allowlist — unset vars or a missing header deny everything (503/403), so deploying before Access exists is safe; POSTs additionally require `Sec-Fetch-Site: same-origin` (CSRF, decision 0006 pattern). Pages are server-rendered HTML, no client framework: `/` dashboard, `/dealers` (incl. minting 1-hour single-use password-reset links whose confirm half is the live `functions/api/auth/password-reset.ts`), `/listings` moderation, `/slots` featured-slot contracts, `/social` boost-queue oversight, `/ops` (table row counts, market-sync freshness, rate-limit clearing, stale pending-media purge, audit-trail view). Every mutation is written to `admin_audit_log` (migration 0017, `at` unix seconds) via `audit()` in src/lib/audit.ts. One-time Zero Trust setup and post-setup verification steps: docs/runbook.md §"Admin panel".

## Invariants

- Neither standalone Worker is deployed by `npm run deploy`/CI (which only ship dist/ + functions/): after any change, `cd workers/expire-sweeper && npx wrangler deploy` and/or `cd workers/admin && npx wrangler deploy` separately.
- The cron dispatch in workers/expire-sweeper/src/index.ts is exact-match on the trigger string; the four constants (SWEEP_CRON, MARKET_SYNC_CRON, WEEKLY_REPORT_CRON, MONTHLY_REPORT_CRON) must stay in sync with `[triggers].crons` in its wrangler.toml — an unmatched cron throws.
- `market_stats` must never feed a public surface (privacy invariant, migrations 0016/0019 headers): marketplace asking prices sit below dealer retail; the table is read only by GET /api/listings/:id/stats behind the marketAnalytics entitlement and by Pro report hints. Seller segments (dealer/private/unknown) must not be mixed — the PK includes seller_kind (0019 rebuild).
- Money is INTEGER cents everywhere; the market view emits whole dollars and syncMarketStats multiplies by 100. Timestamps (synced_at, sent_at, admin_audit_log.at) are unix seconds.
- One report per (period, dealer): the report_runs PRIMARY KEY (period, dealer_id) + INSERT OR IGNORE reservation is the idempotency mechanism; period keys are Monday-anchored (weekly) / month-keyed (monthly) so re-runs on other days reproduce the same key.
- REPORTS_UNSUB_SECRET must be byte-identical on the cron Worker (mints links, reports.ts hmacHex over `reports-unsub:v1:<dealer_id>`) and the Pages project (verifies, functions/api/reports/unsubscribe.ts) — rotating one side silently breaks every previously sent unsubscribe link.
- The admin Worker fails closed at every layer: unset ACCESS_* vars → 503, missing/invalid Access JWT → 403, email outside ADMIN_EMAILS → 403, cross-origin POST → 403. Never weaken to "Access in front is enough" — the in-Worker check is what protects a misconfigured/deleted Access application.
- CSP script-src carries NO 'unsafe-inline'. Every executable inline script must be either build-time-hashed (SSG, via generate-csp-hashes.mjs) or nonce-stamped (dynamic, via takeCspNonce). An inline on*= event-handler attribute anywhere in dist FAILS the build — never add one.
- Unique inline-script hash count must stay <= MAX_HASHES (64); a blowout means a component is interpolating per-page values into an inline script and must be refactored to data-* attributes.
- audit:seo AND audit:launch must pass before any deploy — both run in the local predeploy hook and as explicit CI steps (wrangler-action bypasses the npm predeploy hook, audit #19).
- Attaching the japanauto.ca custom domain opens indexing immediately — public/robots.txt is already Allow-all; the Disallow-all robots is only synthesized by middleware for *.pages.dev. All demo/test data must be removed before domain attach.
- Branch names must reach wrangler-action via the DEPLOY_BRANCH env var, never spliced directly into the command string (injection guard, audit #37).
- deploy.yml must auto-deploy on `main` ONLY (plus the `schedule:` cron and manual `workflow_dispatch`). Never re-add `feature/**`/`fix/**` push triggers: preview Functions bind prod D1/KV/R2 and the public `*.pages.dev` host exposes unauthenticated prod-D1 write paths, so a branch auto-deploy is a prod-write surface (deep-audit OPS-1).
- The scheduled/main-push catalog refresh only tracks D1 if the deploy token has `D1:read` on japanauto-prod. Without it, `export-catalog-data.mjs` fails and the fail-safe (`… || echo "::warning::…"`) rebuilds from the committed catalog-live.json snapshot — so browse pages/counts silently stop tracking new inventory while every scheduled run still succeeds. Treat a persistent "catalog export from D1 failed" CI warning as a broken freshness pipeline (deep-audit ADV-1).
- JWT_SECRET must be a string >= 32 chars; auth fails closed otherwise (verifyAccessToken denies all tokens, signing throws) — a prod deploy without a valid secret breaks auth entirely rather than silently bypassing it.
- D1 database_id b0d65b95-2f43-403d-9237-0d4cac6e186a (japanauto-prod) is referenced in THREE wrangler.toml files (root, workers/expire-sweeper, workers/admin) and must stay in sync.
- Rate-limit checks fail closed: a D1 error propagates to a 5xx rather than allowing the request.
- Public-by-design vars (CLOUDFLARE_ACCOUNT_ID, PUBLIC_CLOUDFLARE_ACCOUNT_HASH, INDEXNOW_KEY, MARKET_SUPABASE_URL, ACCESS_*) may live in wrangler.toml; true secrets (JWT_SECRET, STRIPE_*, RESEND_API_KEY, DAILY_IP_HASH_SALT, CLOUDFLARE_IMAGES_API_TOKEN, SOCIAL_FACTORY_TOKEN, META_FEED_KEY, REPORTS_UNSUB_SECRET, MARKET_SUPABASE_SECRET_KEY, MARKET_SYNC_JWT, ANTHROPIC_API_KEY) must only be set via wrangler secret put / dashboard and never committed.

## Design decisions

- **Run ALL scheduled jobs in the single existing cron Worker, dispatched by exact cron-expression match.** — Pages Functions cannot register scheduled() handlers; rather than one Worker per job (four deploy lifecycles, duplicated D1 binding + secret config), the historic expire-sweeper carries every trigger and `scheduled()` switches on `controller.cron`. Exact match both ways: an unmatched string (e.g. a trigger reformatted in the dashboard) throws instead of silently running the wrong job. *Rejected:* per-job Workers (operational overhead); time-of-day heuristics for dispatch (silent misfires).
- **Market-sync authenticates via a fallback ladder, least privilege first.** — Rung 1: sb-secret apikey + a `{"role":"japanauto_sync"}` HS256 JWT in Authorization, narrowing Postgres rights to the stats view only. Rung 3 fallback: the Secret API key alone (service-role rights — broader than designed, accepted because it lives only in Cloudflare secrets; the scraper project runs ES256 signing keys, so the originally-planned self-validated HS256 JWT could not be guaranteed). A rejected rung logs loudly and falls through; all rungs rejected → throw. A daily sync must not die over an auth experiment. *Rejected:* service-role-only (no privilege narrowing when the JWT does work); hard-failing on the first 401 (turns an upstream key rotation into a dead pipeline).
- **Admin panel as a separate Worker behind Cloudflare Access** — see docs/decisions/0014. Infra consequences here: a third deployable with its own lifecycle, a custom_domain route, preview_urls off, and ACCESS_* vars committed as non-secrets.
- **Run the TTL-expiry sweep as a standalone cron Worker (workers/expire-sweeper) instead of inside the Pages project.** — Cloudflare Pages Functions cannot register scheduled() handlers; a cron trigger requires a real Worker. It shares the same prod D1 and the sweep is idempotent. *Rejected:* Doing expiry lazily on read only (rejected: read-side already filters expired rows, but status stayed 'active' forever, skewing dealer dashboards and analytics — audit #8). A Durable Object alarm (heavier than needed).
- **Drop 'unsafe-inline' from CSP script-src and allowlist inline scripts by build-time SHA-256 hash (SSG) plus per-request nonce (dynamic HTML).** — Eliminates the broadest XSS vector while still allowing Astro's inlined scripts; ~24 unique hashes cover 900+ static pages, and the build gate prevents regressions (on*= handlers, hash-count blowups). *Rejected:* Keeping 'unsafe-inline' (rejected: defeats CSP). Using only a nonce everywhere (rejected: SSG pages are statically cached and cannot carry a per-request nonce). 'unsafe-hashes' for event handlers (rejected: handlers refactored to addEventListener instead).
- **Duplicate the SEO/launch audits as explicit CI steps rather than relying on the npm predeploy hook.** — cloudflare/wrangler-action invokes `pages deploy` directly and never runs the npm `predeploy` lifecycle hook, so CI could ship SEO-broken or demo-marked HTML a local `npm run deploy` would reject (audit #19). *Rejected:* Trusting the predeploy hook in CI (rejected: bypassed by wrangler-action). Making deploy a single npm script in CI (not done).
- **Move rate-limit counters from KV to D1 (table rate_limits, migration 0008) using atomic INSERT...ON CONFLICT...RETURNING.** — The KV version did get->put with no compare-and-swap, so parallel bursts all read the same count and bypassed limits. SQLite's write lock serializes check-and-increment, giving each concurrent request a distinct post-increment count. *Rejected:* KV with optimistic concurrency (rejected: KV has no CAS). A Durable Object counter (rejected: heavier; D1 already present). Cloudflare's native Rate Limiting (not adopted).
- **Keep the IMAGES Workers binding declared in env.d.ts but route all image work through the Cloudflare Images REST API + imagedelivery.net.** — The wrangler.toml [images] block is unsupported by the Pages CLI (must be set in the dashboard); upload-url mints Direct Upload URLs via CLOUDFLARE_IMAGES_API_TOKEN and browsers fetch variants from imagedelivery.net/<hash>. *Rejected:* Using the runtime IMAGES binding for transforms (declared but currently unwired — effectively a skeleton).

## Security notes

- Secrets are documented only as comments in wrangler.toml and typed in env.d.ts; actual values come from `wrangler pages secret put` / Pages dashboard / per-Worker `wrangler secret put` and GitHub Actions secrets (CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID). .gitignore excludes .env*, .dev.vars*, .wrangler/ and the *.local secret copies.
- wrangler.toml has 0600 perms but contains live public identifiers (CLOUDFLARE_ACCOUNT_ID, account hash, INDEXNOW_KEY, prod D1 id) — public-by-design but it does expose infra topology if the repo is shared.
- CI deploy.yml auto-deploys on push to `main` ONLY (plus a 3-hourly `schedule:` cron and manual `workflow_dispatch`); the `feature/**`+`fix/**` push triggers were removed (deep-audit OPS-1) because a preview deployment's Functions bind the prod D1/KV/R2 and the public `*.pages.dev` host exposes unauthenticated prod-D1 write paths — a branch auto-deploy was a live prod-write surface (pages.dev is reachable independent of the unattached apex). A one-off branch deploy is now an explicit `workflow_dispatch`. Branch-name injection into the deploy command was closed earlier (audit #37) by passing via DEPLOY_BRANCH env.
- Middleware enforces HSTS preload, nosniff, X-Frame-Options SAMEORIGIN, frame-ancestors 'self', object-src 'none', a strict CSP, a 64KB /api body cap (audit #33), CSRF rejection of cross-site unsafe-method requests (audit #9), and a /dealer JWT auth guard with a token_epoch session kill-switch (audit #11). The error boundary (audit #23) ensures even thrown handlers get security headers + clean JSON.
- The admin Worker never trusts the edge alone: Access can be misconfigured or its application deleted, so requireAdmin() re-verifies the Access JWT in-Worker against the live JWKS and the email allowlist, fails closed on unset config, and the panel's only unauthenticated response surface is a plain 403/503 string. preview_urls=false keeps version previews off hostnames outside the Access application. Admin errors log stacks to Worker logs only — nothing leaks to the browser.
- Report unsubscribe links are capability URLs: HMAC-SHA256 over a versioned message (`reports-unsub:v1:<dealer_id>`), compared timing-safely; the endpoint is noindex + no-store and authorizes exactly one dealer's opt-out flag. The factory pull API and the Meta feed use the same constant-time key-compare pattern (functions/api/_lib/factory-auth.ts; functions/feeds/meta-vehicles.csv.ts).
- JWT_SECRET fails closed below 32 chars (audit #12): misconfiguration disables auth entirely instead of accepting weakly/unsigned tokens. JWT_SECRET is also reused as the stable salt for hashIpStable (one-way SHA-256, never exposes the secret).
- PII minimization: contact_reveals.ip_hash uses a daily-rotating salt (hashIp); refresh_tokens store hashIpStable (no raw IP) so a D1/backup leak never exposes plaintext dealer IP history (audit #20, PIPEDA/GDPR). The 6h retention batch keeps these tables (and raw-key rate_limits rows) from accumulating.
- Both standalone Workers and `wrangler dev --test-scheduled` operate against the REAL prod D1 (remote=true); there is no local-simulator isolation. The sweep is idempotent, but a locally fired report cron WILL send real mail to every opted-in dealer and mark report_runs (docs/runbook.md warns; dry-run procedure documented there).
- The MEMORY note 'git push does NOT auto-deploy; use npm run deploy' is in tension with the present .github/workflows/deploy.yml, which DOES deploy via wrangler-action on push to `main` (and on the 3-hourly `schedule:` cron). Post-OPS-1 the auto-deploy surface is narrower — only `main` pushes and the schedule, no longer any branch — but pushes to `main` still ship. Whether they actually deploy depends on the CLOUDFLARE_API_TOKEN secret being configured in the GitHub repo — verify before assuming either behavior.
